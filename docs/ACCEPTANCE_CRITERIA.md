# Recruiter contact-record acceptance criteria

## Recruiter workflow

- Capture runs only after the recruiter clicks Capture on a supported active LinkedIn profile or company page. It reads only visible name, title, company, and profile URL fields when available, never infers a company domain or captures contacts, and requires review before the separate Research action.
- A recruiter can use reviewed captured fields or manual prospect details, run configured-source research, and land on a person record. Research fails locally before any source request when an exact identity or first-name + last-name + domain variant matches a saved `Do not contact` record. A recruiter-entered person email/phone is rejected unless the normalized prospect has a canonical LinkedIn `/in/` URL or a full name plus company domain, and the API enforces the same rule before and after asynchronous enrichment.
- A recruiter can choose a recruiter-controlled CSV from the Shortlist, download a schema template, and preview up to 1,000 rows/1 MB before any addition or update. Parsing stays in the extension and never automatically invokes Hunter or company-page lookup; a known provider opt-out may trigger an immediate safety deletion before confirmation.
- CSV intake accepts only allowlisted identity, company, explicitly labeled work-contact, evidence, list, and DNC fields; rejects generic, personal, private, home, mobile, cell, and WhatsApp email/phone columns; supports UTF-8 BOM, CRLF/LF, quoted commas/newlines, and escaped quotes; and visibly reports rejected rows and ignored columns.
- Every valid CSV row has a full name and explicit company domain; findEmm does not infer the domain from an email, and optional `profile_url` accepts only a LinkedIn `/in/` person URL. Before merge, only name, domain, and that person URL go to the authenticated loopback API for a durable HMAC suppression check. A matching person opt-out is excluded, matching saved plaintext is deleted only by the alias type reported as matched, and an unavailable or invalid screen fails closed.
- Imported work emails and recruiter-asserted business phones retain file/row provenance, remain `Recruiter supplied — identity unchecked`, and cannot enable drafting. Provider, confidence, verification, and confirmation claims in a CSV are ignored.
- CSV preview reports additions, updates, named/row-numbered conflicts, DNC safeguards, stripped contacts, provider opt-outs, immediate safety deletions, and provider-blocked domains. Exact identity keys with consistent names may merge, stronger local contacts and context survive, existing DNC remains sticky, and an ambiguous incoming DNC blocks confirmation. Confirmation rechecks the suppression store; any change refreshes the preview and requires another confirmation. A stable confirmation performs a revision-checked encrypted-vault write for eligible additions, then screens again; final-screen failure rolls additions back and a new match is purged.
- The record presents every contact with value, type, status, source/evidence link when available, retrieval date, confidence, and plain-language reason.
- Company-page contacts are marked company-level, retain a source excerpt, and are never treated as person-specific draft addresses.
- Public company channels, recruiter-supplied contacts, fresh provider-valid mailboxes, stale/unverified provider candidates, shared candidates, and locally confirmed person matches have distinct visible treatments.
- Hunter mailbox validity is never presented as proof of person ownership. A `valid` result is fresh only when its provider verification date is no more than 90 days old; missing, future, or older dates display an unverified or stale state.
- No email is draftable until the recruiter explicitly confirms the person match locally. Company channels cannot be confirmed as person matches. The resulting label states that confirmation is a local recruiter attestation, not proof of consent.
- A recruiter can save a record, move it among local lists, save a note, queue a draft-only follow-up, export the record as CSV, and open a mail client draft only for a locally confirmed work email.
- A person record recommends up to three relevant saved recruiting contacts using local company/domain, title, seniority, decision-maker, and former-company successor signals; it explains each match and excludes `Do not contact` records.
- Repeated saves of the same profile update one record and preserve existing contacts, notes, list placement, and queued follow-ups.
- When repeat profile research shows a changed title, company, or domain, the prior employment snapshot stays visible on the local record.
- Moving a record to `Do not contact` disables draft and follow-up actions and keeps it out of recommendations.
- The Shortlist view searches and filters saved people, shows active/follow-up/Do-not-contact counts plus queued-draft and role-change context, and opens a selected local record.
- A recruiter can confirm deletion of one saved record without clearing unrelated records, or separately clear the entire encrypted vault.
- A recruiter can export exactly one selected local list as a passphrase-protected handoff file. Unrelated opt-out identities, local API pairing tokens, notes, and queued drafts are excluded.
- Import decrypts and validates the file, suppression-screens every record before preview and again at confirmation, then shows additions, exact-profile merges, ambiguous conflicts, blocking DNC conflicts, suppressions, immediate safety deletions, and removed contacts. Known provider opt-outs are excluded; active records without a durable suppression alias are excluded; any unresolved incoming DNC blocks every row in that handoff from being merged. The recruiter must explicitly confirm eligible additions.
- Exact matching LinkedIn person URLs with consistent names deduplicate. Ambiguous name-and-company matches remain conflicts, imported IDs are regenerated, and existing local `Do not contact` state wins while contact fields and queued outreach are removed. A first-name + last-name + domain DNC alias blocks an active middle-name/profile variant; an incoming DNC alias suppresses the matching local record atomically.
- Imported provider-valid, recruiter-confirmed, recruiter-supplied, and other person-specific claims become `Shared contact — recheck required`; a new local confirmation is required before drafting.
- Saving, notes, lists, shortlist state, and follow-up queue state stay in Chrome local storage. The product never sends email or automatically contacts a prospect.

## Safety and release gates

- The local API accepts only an authenticated pairing token and listens on `127.0.0.1`.
- Company-page lookup is off by default. Enabling it requires explicit operator approval and an exact-domain allowlist; requests are robots-aware, host-spaced, unauthenticated, HTTPS-only, and response-capped. Source restrictions, breach data, generated person-email guesses, and personal-phone guessing are out of scope.
- Hunter Email Finder is off by default, requires a user-owned key plus explicit confirmation of written commercial approval, sends the key only in a header, caches in memory, and rate-gates calls below provider limits.
- Every Hunter route requires a full name and company domain locally; handle requests still persist both keyed handle and name/domain suppression aliases, and Capture does not infer the missing domain.
- Hunter `valid` plus a verification date within 90 days receives `Provider-valid mailbox — identity unconfirmed`; `accept_all`, `unknown`, missing dates, future dates, and stale dates remain non-draftable candidates.
- A provider `451` or stored person match returns no contact, stores only keyed HMAC aliases, blocks future matching provider access, and purges matching cleartext local identity, contacts, and queued outreach. The API checks durable person suppression before enrichment and again after asynchronous enrichment, before returning a result or persisting a batch. A blocked domain stores only a keyed domain HMAC and blocks later provider access for that domain. Corrupt or unavailable suppression storage disables Hunter rather than bypassing the safeguard.
- The authenticated suppression-screen route accepts at most 1,000 normalized identity records, performs no provider or website lookup, returns only `index`, `checkable`, `suppressed`, `matchedLinkedIn`, `matchedPerson`, and `blockedDomain` booleans/indexes (never HMAC values), and does not persist submitted cleartext identities.
- Encrypted-vault writes and full deletion are serialized in the extension background worker and compare an expected storage revision; a stale popup cannot overwrite a newer encrypted vault.
- A changed vault revision invalidates read-only Research, export, confirmation, queue, and draft actions in other open popups until they unlock again. Every unlock screens all saved records against durable suppression aliases in batches of at most 1,000 before enabling those actions; a failed screen or failed purge leaves the decrypted popup quarantined until another unlock reconciles it. Active uncheckable legacy records are quarantined from contact and export actions while a separate Research action remains available to add the missing domain or LinkedIn identity. A valid matched result is remembered before preview-freshness checks and blocks matching actions plus full handoff export while deletion is pending. Suppression checks remain point-in-time across the separate server and Chrome stores; tests and documentation must not claim a fully atomic cross-store snapshot.
- The local server retains at most 20 transient batches. Each batch expires after one hour, restart, or earlier capacity eviction.
- Handoff imports are capped at 5 MB and 1,000 sanitized records. A wrong passphrase, changed file, unsupported version, or invalid payload fails without changing the local vault.
- Handoff UI states that only the selected list is exported and that the feature provides encrypted files, not hosted sync, user identity, role-based access control, revocation, audit logging, or sender verification; it warns users to transfer the file and passphrase separately and delete the downloaded file when finished.
- `npm test`, `npm run check`, `npm run build`, MV3 manifest validation, and authenticated local API smoke tests must pass.
- A real Chrome smoke covers consent, clicked capture and review, template download, synthetic CSV preview/confirm/immediate suppression purge, post-write failure rollback, imported-record Research, Hunter-disabled durable-opt-out blocking, local confirmation before draft, shortlist search/filter/open/delete, suppression-screened handoff downgrade/purge, stale cross-popup read/write rejection, CSV export, and full-vault clear without real personal data.
- Browser visual verification is required for final release signoff; if a Chrome automation surface is unavailable, that gap must remain reported rather than claimed as completed.
