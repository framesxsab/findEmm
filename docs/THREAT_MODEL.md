# Threat model

## Assets

Captured prospect fields, recruiter inputs, recruiter-controlled source CSV files, source evidence, shortlist records, exported files, share passphrases, provider keys, suppression HMACs, and the local pairing token are sensitive. No asset is intentionally sent to a findEmm-operated service.

## Controls

- Server binds to `127.0.0.1`, requires `x-findemm-token`, and rejects other origins.
- Browser CORS permits only Chrome-extension origins; command-line requests without an Origin header still require the pairing token.
- Token is randomly generated on first run in ignored `server/data/config.json`; extension storage retains it only on the local browser profile.
- Active-page capture requires a click, accepts supported LinkedIn profile/company URLs, returns only allowlisted visible fields, and requires recruiter review before Research. No persistent content script or background collection is used.
- Recruiter CSV parsing is explicit and local to the extension, capped at 1 MB/1,000 rows/50 columns, uses an allowlisted schema and bounded fields, rejects malformed quoting plus generic/personal/private/home/mobile/cell/WhatsApp contact columns, validates explicitly labeled work contacts, domains, LinkedIn `/in/` person URLs and HTTP(S) evidence URLs, and ignores untrusted provider/verification/confirmation columns.
- CSV rows require a full name and explicit company domain so the durable person HMAC alias is always checkable; the domain is never inferred from an email. Before merge, only name, domain, and a canonical LinkedIn person URL go to an authenticated loopback route that compares derived aliases with the durable HMAC store in one batch, performs no provider or website call, returns only `index`, `checkable`, `suppressed`, `matchedLinkedIn`, `matchedPerson`, and `blockedDomain` values, never returns HMACs, and retains no request batch. A failed or uncheckable CSV screen prevents merge.
- CSV additions require a visible preview and confirmation. Provider-opt-out rows are excluded and alias-matched saved plaintext is deleted immediately; the response distinguishes person-alias and LinkedIn-alias matches so a mixed row cannot purge an unrelated URL identity. Existing DNC and stronger local contact claims survive, first-name + last-name + domain variants cannot bypass DNC by adding a middle name or alternate profile URL, incoming exact DNC removes contacts/queue state, and ambiguous incoming DNC blocks the entire confirmation. The local suppression screen runs again at confirmation; changed results refresh the preview and require a second confirmation. This narrows but cannot eliminate a distributed race if suppression state changes after the final snapshot. Imported contacts remain identity-unchecked and non-draftable.
- Only canonical LinkedIn `/in/` handles are stable URL identity keys. A matching URL with a different normalized name is a conflict, and first/last-name DNC ambiguity fails closed unless distinct canonical profile URLs disambiguate the people.
- Optional Hunter credentials exist only in the local server process environment, are sent in `X-API-KEY`, and never enter Chrome storage, URLs, logs, or API responses.
- Hunter remains off without both a user-owned key and operator confirmation of written commercial approval. Consent names Hunter and the transmitted fields.
- A fresh provider-valid mailbox requires Hunter `valid` plus a verification date no more than 90 days old. It remains an identity-unconfirmed person candidate; only a separate local recruiter confirmation makes an address draftable. Recruiter-entered person contacts and any draft/export action also require a canonical LinkedIn `/in/` identity or a full name plus valid company domain so the durable suppression screen is checkable.
- Provider opt-outs are checked before lookup and immediately after asynchronous enrichment, before a response or transient batch is released, and retained only as keyed SHA-256 HMAC values in the local suppression file. Suppression storage fails closed; cleartext matching extension records, contacts, and queued outreach are purged.
- Company-page lookup is off by default and requires explicit approval plus an exact-domain allowlist. Enabled requests apply HTTPS, agent-specific robots rules, response caps, host spacing, and no authentication.
- Extension uses minimum MV3 permissions and displays source evidence/status.
- Encrypted-vault writes and deletion are serialized in the existing background worker and compare a monotonic storage revision. A stale popup aborts instead of overwriting a newer vault and revision-change listeners disable read-only outreach/export actions until re-unlock; encryption still happens in the popup and the passphrase is not sent to the worker. Unlock screens saved records in batches of at most 1,000 and leaves the popup quarantined if screening or purge persistence fails. Active uncheckable legacy records are individually quarantined from contact/export while Research remains available to repair the missing domain or LinkedIn identity. Valid matched signals are recorded before preview-freshness checks, so matching actions and whole-list handoff export stay blocked even if a preview becomes stale or a purge write fails.
- Hunter calls are memory-cached, in-flight deduplicated, and serialized below the documented provider rate limit. Provider opt-outs stop enrichment before company-page lookup.
- Transient batch responses are memory-only, expire after one hour or restart, and are capped at 20 batches with oldest-first capacity eviction.
- Team handoffs use a versioned, authenticated AES-GCM envelope and a transient share passphrase. The encrypted payload is allowlisted and excludes the local API pairing token, notes, and queued drafts.
- Handoff import caps file and record counts, sanitizes fields and URLs, regenerates external IDs, and uses the authenticated local suppression screen before preview and again at confirmation. Known provider matches are excluded and alias-matched saved plaintext is deleted immediately; active records without a durable suppression alias fail closed and are excluded. Any unresolved incoming DNC is a blocking conflict for the entire handoff. Eligible additions require explicit confirmation.
- Only the explicitly chosen list is exported; unrelated opt-out identities are excluded. Imported person-specific claims are downgraded for local recheck. Exact LinkedIn person URLs with consistent names may merge automatically, ambiguous identity matches do not, and an existing local `Do not contact` state remains sticky across first-name + last-name + domain variants.
- The searchable local shortlist exposes saved-state counts and supports confirmation-gated deletion of one record or deletion of the full encrypted vault.
- Users are warned to send the file and passphrase through separate channels and to delete the downloaded file after use.

## Non-goals

This is not a contact-data vendor, live ATS/CRM connector, consent-management platform, identity-verification service, email delivery service, or guarantee that any contact detail is current or lawful to use. The recruiter remains responsible for the source CSV, permission to process its data, and deletion of the plaintext file. Mailbox validity does not prove person ownership; recruiter confirmation does not prove outreach consent. Handoffs are not hosted sync and provide no user identity, role-based access control, revocation, audit logging, or sender verification. Anyone with both the file and passphrase can decrypt it, and findEmm cannot revoke a copied file. Suppression HMACs reduce retained plaintext but remain sensitive if the separate local suppression secret or config file is compromised. Suppression screens are point-in-time: confirmation, post-write reconciliation, and unlock-time reconciliation reduce the interval, but a new server suppression is not transactionally coupled to the Chrome vault revision and can occur after the final screen.
